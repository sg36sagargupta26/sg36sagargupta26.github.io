Designing a Rate Limiter in Spring Boot — From Naive to Production-Ready

June 14, 2026 • 12 min read • Spring Boot, LLD, Rate Limiting, Redis, Concurrency

Rate limiting is one of those features that starts deceptively simple — a counter, a map, maybe a cron job. But as you peel the layers, you hit concurrency, race conditions, distributed state, and performance tradeoffs. This post walks through the evolution of a rate limiter in Spring Boot, mirroring what a real LLD interview might look like: start naive, get challenged, iterate.

We'll implement it as a HandlerInterceptor (the Spring Boot way), though a Servlet Filter works equally well.

V1: Naive Counter with Cron Reset

Our first instinct: store a count per user in a ConcurrentHashMap, reset every minute with a scheduled task.

@Component
public class RateLimitInterceptor implements HandlerInterceptor {

    private final ConcurrentHashMap counts = new ConcurrentHashMap<>();
    private static final int MAX_REQUESTS = 100;

    @Override
    public boolean preHandle(HttpServletRequest request,
                             HttpServletResponse response,
                             Object handler) throws Exception {

        String userId = request.getHeader("X-User-Id"); // or IP
        int current = counts.merge(userId, 1, Integer::sum);

        if (current > MAX_REQUESTS) {
            response.setStatus(429);
            response.getWriter().write("{\"error\":\"Too Many Requests\"}");
            return false;
        }
        return true;
    }

    @Scheduled(fixedRate = 60_000)
    public void resetCounts() {
        counts.clear();
    }
}

The Problem

At second 59 of minute N, a user sends 100 requests. At second 1 of minute N+1, the map clears and they send another 100. That's 200 requests in ~2 seconds — the rate limiter failed.

Design lesson: A fixed-window reset creates a boundary condition where 2× the limit passes in a burst. This is why sliding windows exist.

V2: Sliding Window with Timestamp List

Instead of just a count, we store the list of timestamps per user. On each request, we clean old entries and check the window size.

@Component
public class RateLimitInterceptor implements HandlerInterceptor {

    private final ConcurrentHashMap> timestamps = new ConcurrentHashMap<>();
    private static final int MAX_REQUESTS = 100;
    private static final long WINDOW_MS = 60_000;

    @Override
    public boolean preHandle(HttpServletRequest request,
                             HttpServletResponse response,
                             Object handler) throws Exception {

        String userId = request.getHeader("X-User-Id");
        long now = System.currentTimeMillis();
        long cutoff = now - WINDOW_MS;

        List userTimestamps = timestamps.computeIfAbsent(
            userId, k -> new ArrayList<>()
        );

        // Remove expired timestamps
        userTimestamps.removeIf(ts -> ts < cutoff);
        userTimestamps.add(now);

        if (userTimestamps.size() > MAX_REQUESTS) {
            response.setStatus(429);
            response.getWriter().write("{\"error\":\"Too Many Requests\"}");
            return false;
        }
        return true;
    }

    @Scheduled(fixedRate = 60_000)
    public void cleanupOldUsers() {
        long cutoff = System.currentTimeMillis() - WINDOW_MS;
        timestamps.entrySet().removeIf(entry ->
            entry.getValue().isEmpty() ||
            entry.getValue().get(entry.getValue().size() - 1) < cutoff
        );
    }
}

Now a user genuinely limited to 100/min can't burst 200. The cleanup cron removes stale users from memory.

The Problem

Two threads for the same user can both read, modify, and write the list simultaneously — classic check-then-act race condition. Under 1000 concurrent requests, the list might miss counts or double-count.

V3: Thread Safety with Synchronized Blocks

Add a synchronized block around list access per user:

@Override
public boolean preHandle(HttpServletRequest request,
                         HttpServletResponse response,
                         Object handler) throws Exception {

    String userId = request.getHeader("X-User-Id");
    long now = System.currentTimeMillis();
    long cutoff = now - WINDOW_MS;

    List userTimestamps = timestamps.computeIfAbsent(
        userId, k -> new ArrayList<>()
    );

    synchronized (userTimestamps) {
        userTimestamps.removeIf(ts -> ts < cutoff);
        userTimestamps.add(now);

        if (userTimestamps.size() > MAX_REQUESTS) {
            response.setStatus(429);
            response.getWriter().write("{\"error\":\"Too Many Requests\"}");
            return false;
        }
    }
    return true;
}

The Problem

If we used a CopyOnWriteArrayList, the synchronized block becomes redundant — CoW lists are already thread-safe for reads and writes. But CopyOnWriteArrayList copies the entire array on every write, which is expensive under high throughput. For this use case, a plain ArrayList (or ArrayDeque) under a lock is better. We switch to ArrayDeque for efficient head removal:

private final ConcurrentHashMap> timestamps = new ConcurrentHashMap<>();

// Inside the synchronized block:
ArrayDeque deque = timestamps.computeIfAbsent(userId, k -> new ArrayDeque<>());
synchronized (deque) {
    while (!deque.isEmpty() && deque.peek() < cutoff) {
        deque.poll();
    }
    deque.add(now);
    if (deque.size() > MAX_REQUESTS) { /* 429 */ }
}

V4: Multi-Instance Consistency

So far, the rate limiter lives in application memory. If you deploy 3 instances of your Spring Boot service behind a load balancer, each has its own map. A user can hit instance-1 100 times, then instance-2 100 times — 300 requests slip through.

First instinct: replace the in-memory map with a database table:

CREATE TABLE rate_limits (
    user_id VARCHAR(64) NOT NULL,
    request_time BIGINT NOT NULL,
    INDEX idx_user_time (user_id, request_time)
);

-- Cleanup every hour
DELETE FROM rate_limits WHERE request_time < ?;
// In the interceptor:
@Autowired
private JdbcTemplate jdbcTemplate;

@Override
public boolean preHandle(HttpServletRequest request, ...) {
    String userId = request.getHeader("X-User-Id");
    long now = System.currentTimeMillis();
    long cutoff = now - WINDOW_MS;

    // Clean old entries for this user
    jdbcTemplate.update(
        "DELETE FROM rate_limits WHERE user_id = ? AND request_time < ?",
        userId, cutoff
    );

    // Count remaining
    Integer count = jdbcTemplate.queryForObject(
        "SELECT COUNT(*) FROM rate_limits WHERE user_id = ?",
        Integer.class, userId
    );

    if (count != null && count >= MAX_REQUESTS) {
        response.setStatus(429);
        return false;
    }

    // Insert this request
    jdbcTemplate.update(
        "INSERT INTO rate_limits (user_id, request_time) VALUES (?, ?)",
        userId, now
    );
    return true;
}

The Problem

Every request now hits disk I/O — a DELETE, a SELECT COUNT, and an INSERT. This adds 10-50ms latency per request. For a rate limiter (which should be near-zero overhead), this is unacceptable. Plus, the DB becomes a bottleneck under high traffic.

Design lesson: Databases are for durable state, not transient, high-frequency counters. When your hot path hits disk, you've chosen the wrong tool.

V5: Redis — In-Memory, Distributed, with Native TTL

Redis is the right tool here: in-memory = fast, distributed = multi-instance consistent, and it has built-in TTL and data structures purpose-built for rate limiting.

Approach: store a Sorted Set per user where the score is the timestamp and the member is a unique request ID.

@Component
public class RedisRateLimitInterceptor implements HandlerInterceptor {

    private final StringRedisTemplate redis;
    private static final int MAX_REQUESTS = 100;
    private static final long WINDOW_SECONDS = 60;

    public RedisRateLimitInterceptor(StringRedisTemplate redis) {
        this.redis = redis;
    }

    @Override
    public boolean preHandle(HttpServletRequest request,
                             HttpServletResponse response,
                             Object handler) throws Exception {

        String userId = request.getHeader("X-User-Id");
        String key = "rate_limit:" + userId;
        long now = System.currentTimeMillis();
        long cutoff = now - (WINDOW_SECONDS * 1000);
        String member = now + ":" + UUID.randomUUID().toString().substring(0, 8);

        // Lua script to do everything atomically in Redis
        String script = """
            redis.call('ZREMRANGEBYSCORE', KEYS[1], 0, ARGV[1])
            local count = redis.call('ZCARD', KEYS[1])
            if count >= tonumber(ARGV[3]) then
                return 0
            end
            redis.call('ZADD', KEYS[1], ARGV[2], ARGV[4])
            redis.call('EXPIRE', KEYS[1], ARGV[5])
            return 1
            """;

        Long result = redis.execute(
            new DefaultRedisScript<>(script, Long.class),
            List.of(key),
            String.valueOf(cutoff),
            String.valueOf(now),
            String.valueOf(MAX_REQUESTS),
            member,
            String.valueOf(WINDOW_SECONDS + 10)
        );

        if (result != null && result == 0L) {
            response.setStatus(429);
            response.getWriter().write("{\"error\":\"Too Many Requests\"}");
            return false;
        }
        return true;
    }
}

Why a Lua Script?

Redis operations are atomic individually, but we need multiple operations to be atomic together: remove expired entries → count → check limit → add. A Lua script runs as a single atomic unit on the Redis server — no race conditions, even across instances.

Why Sorted Set (ZSET)?

ZSET gives us ZREMRANGEBYSCORE (remove by timestamp range in O(log N)) and ZCARD (count in O(1)). Together, a sliding window rate check is essentially two O(log N) operations. Much faster than iterating a list.

V6: Production Hardening

A few more considerations for a truly production-ready implementation:

1. Choose the right algorithm

The sliding window log (storing every timestamp) works but uses memory proportional to request count. Alternatives:

Algorithm Memory Accuracy When to Use
Fixed Window O(1) Poor (boundary bursts) Never in production
Sliding Window Log O(requests) Perfect Low-volume APIs
Sliding Window Counter O(1) ~Good (approximate) High-volume APIs
Token Bucket O(1) Good, allows bursts Burst-tolerant APIs

2. Redis as a dependency — what if it's down?

Rate limiting is a defense mechanism, not core business logic. If Redis is down:

  • Fail open: Allow all requests (risk of overload)
  • Fail closed: Block all requests (risk of outage)

In most cases, fail open with degraded local fallback is the right tradeoff — switch to an in-memory map with a warning log.

3. Rate limit by what key?

Not just userId. A production system needs:

  • Per IP: Basic DDoS protection (can be spoofed, but helps)
  • Per API Key / User: Tiered access (free tier = 10/min, premium = 1000/min)
  • Per Endpoint: /login gets stricter limits than /search
  • Global: Absolute cap across all users to protect the entire system

4. Response Headers

Always include rate limit headers so clients can self-regulate:

response.setHeader("X-RateLimit-Limit", String.valueOf(MAX_REQUESTS));
response.setHeader("X-RateLimit-Remaining", String.valueOf(remaining));
response.setHeader("X-RateLimit-Reset", String.valueOf(windowEndEpochSeconds));
