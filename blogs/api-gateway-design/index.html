Designing an API Gateway in Java — From Linear Proxy to Production-Grade

June 14, 2026 • 15 min read • LLD, API Gateway, Service Discovery, Resilience

An API Gateway is the single entry point for all client requests in a microservices architecture. It handles routing, authentication, rate limiting, load balancing, and resilience — all before a request reaches your backend. But most candidates describe it as "a proxy that forwards requests" and stop there. This post walks through the full evolution of an API Gateway design — from a linear, naive proxy to a production-grade system.

V1: The Linear Proxy (MVP Brainstorm)

When asked "design an API Gateway," most engineers start with a feature list. Here's what a typical MVP brainstorming session looks like:

MVP Features (initial pass):
1. Routing           — match path → forward to backend
2. Security          — authentication, identification
3. Rate Limiting     — protect downstream services
4. Observability     — logging, metrics
Interviewer feedback: Good prioritization. Routing first (core). Security early. Rate limiting protects downstream. Observability for ops. This shows production awareness. But you're missing some critical pieces.

What's Missing at V1?

GapWhy It Matters
Load BalancingRouting ≠ load balancing. If service-A has 3 instances, which one gets the request? This is a core gateway responsibility.
Resilience"Consistent error handling" is good, but what about retries? Circuit breaker? Timeouts? These protect the system more than formatted errors.
Observability depthLogging + metrics isn't enough. You need distributed tracing — traceId propagated across all services for end-to-end request visibility.
Security granularity"Security and identification" is too broad. Interviewers expect: authentication (who are you?) vs authorization (what can you do?).

Refined MVP Feature Set

1. Routing + Load Balancing
2. Authentication (JWT / API Key validation)
3. Rate Limiting
4. Observability (logs + metrics + distributed tracing)
5. Resilience (timeouts + retries + circuit breaker)

V2: Understanding the Request Pipeline

The next evolution is understanding that a gateway isn't a single "route → forward" step. It's a pipeline of filters that process the request before and after the backend call.

// Naive mental model (too linear):
Client → Gateway Entry → Route Matching → Backend → Response

// Production mental model (filter chain):
Client
  ↓
Gateway Entry (Netty / Servlet)
  ↓
Pre-Filters (Auth, Rate Limit, Logging)
  ↓
Route Matching
  ↓
Load Balancer + Service Discovery
  ↓
Backend Service Call
  ↓
Post-Filters (Logging, Response Transformation)
  ↓
Response to Client
Key insight: Auth, rate limiting, and logging aren't random features — they're filters in a chain. Each filter has a clear position: before routing (pre-filters) or after the backend response (post-filters). This is exactly how Spring Cloud Gateway and Netflix Zuul are architected.

V3: Building the Core — Round Robin Load Balancer

Let's build a working gateway. We'll start with in-memory backend registration and round-robin load balancing:

@Component
public class BackendServerService {
    private final CopyOnWriteArrayList servers = new CopyOnWriteArrayList<>();

    public void addServer(String url) { servers.add(url); }
    public void removeServer(String url) { servers.remove(url); }
    public List getServers() { return List.copyOf(servers); }
}

@Component
public class RoundRobinLoadBalancer {
    private final AtomicInteger counter = new AtomicInteger(0);
    private final BackendServerService serverService;

    public RoundRobinLoadBalancer(BackendServerService service) {
        this.serverService = service;
    }

    public String next() {
        List servers = serverService.getServers();
        if (servers.isEmpty()) throw new RuntimeException("No backends available");
        int index = counter.getAndIncrement() % servers.size();
        return servers.get(index);
    }
}

And a simple routing filter that intercepts requests:

@Component
public class GatewayFilter implements Filter {

    private final RoundRobinLoadBalancer lb;
    private final RestTemplate restTemplate = new RestTemplate();

    @Override
    public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain)
            throws IOException, ServletException {

        HttpServletRequest request = (HttpServletRequest) req;
        HttpServletResponse response = (HttpServletResponse) res;

        String path = request.getRequestURI();
        String backend = lb.next();

        try {
            ResponseEntity backendResp = restTemplate.getForEntity(
                backend + path, String.class
            );
            response.setStatus(backendResp.getStatusCodeValue());
            response.getWriter().write(backendResp.getBody());
        } catch (Exception e) {
            response.setStatus(502);
            response.getWriter().write("{\"error\":\"Bad Gateway\"}");
        }
    }
}
This works for a demo. But here's what's wrong:
IssueWhy It Fails
Static server listHardcoded addresses. What happens when instances scale up/down?
No health checksDead backends still get traffic. Users hit 502 errors.
No weightingAll instances treated equal. A 2-core instance gets same load as 16-core.
Not distributedEach gateway instance has its own counter. No global load distribution.
Blocking I/OServlet + RestTemplate = one thread per request. At 10K concurrent, you run out of threads.
No resilienceNo timeout, no retry, no circuit breaker. A slow backend blocks the thread pool.

V4: Dynamic Backend Management + Service Discovery

Static server lists are a non-starter in production. We need dynamic registration — backends join and leave the pool at runtime.

@RestController
@RequestMapping("/admin")
public class AdminController {

    private final BackendServerService serverService;

    @PostMapping("/servers")
    public ResponseEntity addServer(@RequestBody ServerRequest server) {
        serverService.addServer(server.getUrl());
        return ResponseEntity.ok("Server added: " + server.getUrl());
    }

    @DeleteMapping("/servers")
    public ResponseEntity removeServer(@RequestBody ServerRequest server) {
        serverService.removeServer(server.getUrl());
        return ResponseEntity.ok("Server removed: " + server.getUrl());
    }
}

V5: Adding Security — But Who Owns Auth?

We add authentication to the filter chain. JWT validation happens before the request reaches any backend:

@Component
public class JwtAuthFilter implements Filter {

    private final JwtValidator jwtValidator;

    @Override
    public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain)
            throws IOException, ServletException {

        HttpServletRequest request = (HttpServletRequest) req;
        HttpServletResponse response = (HttpServletResponse) res;

        String authHeader = request.getHeader("Authorization");
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            response.setStatus(401);
            return;
        }

        String token = authHeader.substring(7);
        if (!jwtValidator.validate(token)) {
            response.setStatus(403);
            response.getWriter().write("{\"error\":\"Invalid token\"}");
            return;
        }

        chain.doFilter(req, res);
    }
}
Critical distinction: The gateway should validate tokens, not issue them. Token issuance belongs in a separate Auth Service. The gateway is a security enforcement point, not an identity provider. Mixing these makes the gateway a monolith.

Filter Order Matters

In Spring, @Order controls the filter chain sequence:

@Order(1)  // JwtAuthFilter        — first, reject unauthenticated
@Order(2)  // RateLimitingFilter   — second, limit authenticated users
@Order(3)  // LoggingFilter (pre)  — third, log the request
// ... route + forward ...
@Order(4)  // LoggingFilter (post) — after response

V6: Resilience Layer — Circuit Breaker, Retry, Timeout

Resilience is where most homegrown gateways fail. Without it, a single slow backend can cascade failure across the entire system.

@Component
public class ResilientGatewayFilter implements Filter {

    private final RoundRobinLoadBalancer lb;
    private final ResilienceConfig config;

    @Override
    public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain) {

        HttpServletRequest request = (HttpServletRequest) req;
        HttpServletResponse response = (HttpServletResponse) res;

        String backend = lb.next();
        int attempts = 0;
        Exception lastError = null;

        while (attempts < config.getMaxRetries()) {
            try {
                // Timeout per attempt
                ResponseEntity resp = restTemplate.exchange(
                    backend + request.getRequestURI(),
                    HttpMethod.GET,
                    null,
                    String.class
                );
                // Success — reset circuit breaker
                circuitBreaker.recordSuccess(backend);
                response.setStatus(resp.getStatusCodeValue());
                response.getWriter().write(resp.getBody());
                return;

            } catch (TimeoutException e) {
                lastError = e;
                attempts++;
                // Exponential backoff
                Thread.sleep((long) Math.pow(2, attempts) * 100);

            } catch (Exception e) {
                lastError = e;
                circuitBreaker.recordFailure(backend);
                // Try next backend
                backend = lb.next();
                attempts++;
            }
        }

        // All attempts failed
        if (circuitBreaker.isOpen(backend)) {
            response.setStatus(503);
            response.getWriter().write("{\"error\":\"Service Unavailable\"}");
        } else {
            response.setStatus(504);
            response.getWriter().write("{\"error\":\"Gateway Timeout\"}");
        }
    }
}

Resilience Patterns Summary

PatternWhat It DoesWhen It Activates
TimeoutAbort request if backend doesn't respond in N msEvery request
RetryRe-attempt on transient failuresTimeout, 5xx errors
Circuit BreakerStop sending requests to a failing backend (fail fast)After N consecutive failures
BulkheadLimit concurrent calls to one backend (isolate failure)Thread pool isolation

V7: Non-Blocking — WebFlux & Netty

A Servlet-based gateway creates one thread per request. At 50K concurrent connections with 5-second backend latency, you'd need 50,000 threads. That's impossible — each thread consumes ~1MB stack space (50GB just for stacks).

The fix: Reactive programming with Spring WebFlux:

@Component
public class ReactiveGatewayFilter implements WebFilter {

    private final ReactiveLoadBalancer lb;
    private final WebClient webClient = WebClient.create();

    @Override
    public Mono filter(ServerWebExchange exchange, WebFilterChain chain) {
        String backend = lb.next();

        return webClient.get()
            .uri(backend + exchange.getRequest().getPath().value())
            .exchangeToMono(clientResponse ->
                clientResponse.bodyToMono(String.class)
                    .map(body -> {
                        exchange.getResponse().setStatusCode(
                            clientResponse.statusCode()
                        );
                        return body;
                    })
            )
            .flatMap(body -> exchange.getResponse()
                .writeWith(Mono.just(
                    exchange.getResponse().bufferFactory().wrap(body.getBytes())
                ))
            )
            .timeout(Duration.ofSeconds(5))
            .retry(3)
            .onErrorResume(e -> {
                exchange.getResponse().setStatusCode(HttpStatus.BAD_GATEWAY);
                return exchange.getResponse().setComplete();
            });
    }
}
Why this works at scale: WebFlux uses an event loop (Netty) with a small, fixed thread pool. Instead of one-thread-per-request, it uses non-blocking I/O — threads never wait. 50K concurrent requests run on ~4 threads. This is the same architecture that powers Spring Cloud Gateway and Netflix Zuul 2.
