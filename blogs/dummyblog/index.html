Building a URL Shortener with Analytics — Architecture Deep Dive

June 14, 2026 • 8 min read • Spring Boot, Redis, MySQL, AWS

URL shorteners seem simple on the surface: take a long URL, generate a short code, redirect. But building one that handles millions of redirects with analytics, caching, and sub-50ms latency is a different beast. This post walks through the architecture I built for a production-grade URL shortener using Spring Boot, Redis, MySQL, and AWS EC2.

System Requirements

  • Generate short, unique codes for any valid URL
  • Redirect from short URL to original URL in under 50ms
  • Track click analytics (referrer, user agent, timestamp, IP)
  • Handle high throughput with Redis caching
  • Deploy on AWS EC2 with CloudWatch monitoring

Short Code Generation

The core challenge is generating unique, collision-free short codes. I used a Base62 encoding approach: take a unique incrementing ID (from MySQL auto-increment or a Snowflake-like ID), encode it to Base62 (0-9, a-z, A-Z), and use the result as the short code.

// Base62 encoding in Java
public String encode(long id) {
    char[] chars = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ".toCharArray();
    StringBuilder sb = new StringBuilder();
    while (id > 0) {
        sb.append(chars[(int) (id % 62)]);
        id /= 62;
    }
    return sb.reverse().toString();
}

Why Base62? It's URL-safe, compact (7 characters = ~3.5 trillion combinations), and doesn't need special encoding. A 7-char code like 3dK9f2A maps to a unique long URL in the database.

Caching with Redis

The redirect path is simple: someone hits /3dK9f2A, the server looks up the long URL, and issues a 301 redirect. But if every request hits MySQL, latency spikes under load.

Redis sits between the API and MySQL:

// Redirect flow
GET /3dK9f2A

1. Check Redis     → cache hit? return 301 (1-5ms)
2. Check MySQL     → found? populate Redis, return 301 (10-30ms)
3. Not found       → return 404

I set Redis TTL to 24 hours for hot URLs and used write-through caching — every new short URL is written to both MySQL and Redis simultaneously. This achieved sub-5ms redirects for cached URLs and sub-50ms even on cache-miss.

MySQL Schema

A clean, denormalized schema optimized for reads:

CREATE TABLE urls (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    short_code VARCHAR(10) UNIQUE NOT NULL,
    long_url TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    expires_at TIMESTAMP NULL,
    INDEX idx_short_code (short_code)
);

CREATE TABLE clicks (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    short_code VARCHAR(10) NOT NULL,
    referrer VARCHAR(512),
    user_agent VARCHAR(512),
    ip_hash VARCHAR(64),
    clicked_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_short_code_clicks (short_code, clicked_at)
);
Separating urls and clicks into different tables keeps the redirect query lightweight. The clicks table can grow to millions of rows without slowing down redirects.

AWS Deployment

Deployed on AWS EC2 (t3.medium) running:

  • Spring Boot fat JAR on port 8080 behind Nginx reverse proxy
  • Redis on the same instance (later migrated to ElastiCache for multi-instance setups)
  • MySQL via RDS for managed backups and replication
  • CloudWatch for request latency, error rate, and cache hit ratio dashboards

The entire stack costs ~$45/month on a single EC2 instance with RDS.

Analytics Dashboard

Every click is logged with referrer, user agent, hashed IP (respecting privacy), and timestamp. A simple Spring Boot endpoint aggregates this data:

  • Click count over time (hourly/daily breakdown)
  • Top referrers — where traffic comes from
  • Browser/OS breakdown — parsed from user agent
  • Geo distribution — rough estimate from IP hash

Key Takeaways

  1. Cache aggressively. Redis took redirect latency from 30ms → 5ms and cut MySQL load by 90%.
  2. Pre-compute IDs. Don't generate short codes on redirect — encode them at creation time and store the mapping.
  3. Separate reads from writes. The clicks table is append-only and never queried during redirect. This isolation is critical.
  4. 301 vs 302. Use 301 (permanent) for analytics — browsers cache it, meaning repeat visitors don't hit your server. But if you need per-click analytics accuracy, use 302 (temporary).
